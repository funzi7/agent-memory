## The bug
CI reported "no working Nitter instance" while two instances returned **HTTP 200**
(privacyredirect, tiekoetter) and xcancel a ParseError. The reader gated on a
brittle `"<rss"`/`"<item"` substring check, so a 200 whose RSS was disabled (HTML
returned) or malformed was discarded — a parseable 200 was treated as failure.

## The fix (parsing, not instance-shuffling)
- **RSS-first, then HTML fallback, per instance**: try `/<account>/rss` then the
  HTML timeline `/<account>` (parsed via lxml XPath, robust to markup variants). The
  first instance that yields ≥1 parseable post wins; **a 200 we can parse is used**.
- **Parse-failure vs access-failure** are now distinct in the log (`rss-parse-error`,
  `rss-empty` vs `rss-http:403`, `rss-net:Timeout`), with the full per-instance
  breakdown in the error string — diagnosable.
- Retweets excluded in both paths (RSS: RT marker / different `dc:creator`; HTML:
  `retweet-header` / `username` ≠ account). `http_get` injectable for offline tests.
