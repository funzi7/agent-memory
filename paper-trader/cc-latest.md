## The bug
KORU (Direxion South Korea 3X) executed a 20:1 forward split (2026-07-15). The engine had
no split handling: `leveraged_momentum_10k` held 2 old shares @ ~$763, saw the ~$40
post-split price WITHOUT multiplying shares ×20, read it as a −95% crash, fired the
crash-stop and sold 2 shares for ~$40 — booking a fake ~−$1,488 realized loss
(`realized_pnl` −1228.4) and leaving cash short of true value.

## PART 1 — generic split detection + in-place rescale (permanent)
- **Immune to real crashes.** For each held ticker, compare the STORED close of the last
  processed date (`state/closes.json`) vs the freshly-FETCHED close for that SAME date
  (`bars`). A same-date value only changes when the provider RE-ADJUSTS the series (a
  split), never from a market move → a real −95% day leaves the prior-day stored close
  unchanged → ratio 1 → no split. Only a CLEAN integer N:1 / 1:N within 1% (and a move ≥
  1.5× / ≤ 0.667× to exclude dividend drift) counts.
- On detection: rescale IN PLACE (shares ×r, avg/high_water ÷r; **cost_basis (total $)
  unchanged**), BEFORE marking/valuation/exits → **zero P&L, no trade, value continuous**,
  so the split day never trips crash/stop/trailing/support exits. Forward + reverse, every
  strategy. Hebrew note. Skipped under `--synthetic` (fabricated bars would false-flag).

## PART 2 — one-time KORU repair (guarded, idempotent)
- Guarded by `state.migrations` (`koru_20for1_split_repair_2026_07_15`) AND a data
  re-verification of the 20:1 ratio; **if data is unavailable it does not apply and does
  not record the id — retries next run, never blind.**
- Reverses EXACTLY from the recorded trades: restores KORU **40 @ $38.1546** (763.0914/20),
  hw **$41.4545** (829.09/20), cost $1527.18; deducts the sale's **$39.54** proceeds from
  cash; removes the fake loss (`gross_loss`→0, `n_trades` 4→3, `realized_pnl`
  −1228.4→**+259.24**); repairs only the impacted equity_history points (07-15/07-16);
  drops the redundant pending KORU BUY (the $100 portfolio's KORU is left to the normal
  path). The erroneous crash SELL is removed from `trades.csv` by `cleanup_migration_trades`
  at STATE-PERSISTENCE time so the trade-log edit and the migration id are always written
  together (a run that repairs in memory but never persists re-repairs next run).
  `combined_*` re-derives automatically.

### Dashboard note (agent-memory side)
