## What this change did
Fixed the daily workflow's **secret→env wiring** so the code actually receives the
keys it reads. `.github/workflows/daily.yml` mapped only TELEGRAM + IBKR secrets into
the "Run daily cycle" step, so newer keys were seen as unset even though they exist
as repo secrets — the live run produced `twitterapi.io read failed:
TWITTERAPI_API_KEY not set`. This rides on the same branch/PR as the twitterapi.io
reader swap. Scoped to `funzi7/paper-trader` only (this cc-latest is the explicit
handoff). No dashboard/data change — workflow + docs only.

## The wiring fix (daily.yml "Run daily cycle" step env)
A repo secret only reaches the code if it is mapped into that step's `env:`. Added,
with env-var names that EXACTLY match what `papertrader/` reads via
`os.environ.get(...)`:
- **`TWITTERAPI_API_KEY`** — twitter_track reader (`papertrader/twitter.py`, actively
  read). This was the confirmed break.
- **`ANTHROPIC_API_KEY`** — the strict post→trade classifier (actively read).
- `OPENAI_API_KEY`, `FINNHUB_API_KEY` — **forward-compat only; NOT consumed by any
  code in this repo yet.** Verified by grep: there is no GPT classifier fallback,
  `data.py` uses keyless yfinance/Stooq, and `screener_track` reads the cross-repo
  YAML shortlist (no direct Finnhub call). Wired now so a future OpenAI-fallback /
  Finnhub path works without re-editing the workflow.

The `workflow_dispatch` inputs (`dry_run` / `force_chart` / `recompute_pending`) and
the commit/publish steps are unchanged. The publish step still maps only
`AGENT_MEMORY_PAT`. No secret values are printed — only `${{ secrets.* }}` wiring.
