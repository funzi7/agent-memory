## What this change did
KORU's real 20:1 forward split (2026-07-15) was mis-read as a −95% crash. The earlier V1
repair (PR #23) addressed one shape; the committed state was then corrupted **again** — a
later run re-fired the fake **2026-07-16** crash SELL and "refilled" KORU on **2026-07-17**
(10k: **170** sh, 100: **1** sh) at ~$16.36 — corrupting **BOTH** `leveraged_momentum_10k`
**AND** `leveraged_momentum_100`. V1 (10k-only, network-re-verified) never applied to this state.

This PR adds a **network-free, fingerprint-guarded, idempotent V2 migration** that reverses
both corrupt ops, plus two operational fixes that were preventing any repair from persisting.

## Root cause of the red scheduled Daily
Run **29787505397** (scheduled, 2026-07-20T23:32:11Z, head `637d0be`) failed with
`requests.exceptions.HTTPError: 400 … /sendMessage` raised at `telegram.send(report)`. The
send raised **after** the cycle computed the day's state, so the process exited non-zero and
the workflow's *commit-state* step (exit-0-gated) **silently discarded the day + any repair**.
The delisted-ticker downloads and the twitter/anthropic/openai billing messages in that log
are pre-existing **non-fatal noise**, not the failure. **No workflow file was edited**
(Claude Code cannot edit `.github/workflows/` here); the fix is code-only.

## The fix
- **`corporate_actions.py`** — KORU full-state repair **V2** (`koru_full_state_repair_v2_2026_07_17`):
  guarded by an **exact committed-state + trade-log fingerprint** (07-17 refill + 07-16 crash
  SELL + original 06-24/07-15 BUYs), **not** yfinance closes. Restores the continuous position
  (10k **40 @ 38.1546**, 100 **1 @ 23.5918**) with **zero split P&L**, preserving the **real**
  AMDL loss ⇒ 10k `realized_pnl` **+20.66**, 100 **0**. Records both V1 + V2 ids. Silent no-op
  on a clean/unrelated state; partial/unknown → no change + `ℹ️` diagnostic. `cleanup_koru_v2_
  trades` removes exactly the 4 corrupt rows at persistence time and corrects the surviving
  AMDL SELL snapshot (→ 2758.13 / 7816.34).
- **`main.py`** — (1) **no-new-bar persistence**: an in-memory migration is persisted on the
  equal-date path WITHOUT running fills, advancing the date, or reprocessing the settled day;
  (2) **non-fatal report send**: a Telegram send failure logs a warning and the run still
  returns 0 with state already persisted (state is the product; Telegram is a side-effect).
