### FIX 1 — SWING strategy now appears automatically (fully data-driven)
- **swing_100 / swing_10k are NOT in `portfolios.json` yet** (the daily cron hasn't
  republished since swing merged). The published file currently carries the 8
  existing strategies × 2 sizes = 16 portfolios.
- The dashboard was already iterating the portfolios object (`activityOrder`,
  `groupByStrategy`, the chart and summary all use `Object.keys`), so there is **no
  hard-coded 8-strategy filter**. Confirmed by injecting a synthetic swing pair into
  a local copy: a swing card, a `strategy.html?id=swing` detail page, and the
  activity ordering **all appeared with zero further changes**.
- Added swing's display metadata so it renders with proper Hebrew labels the moment
  it publishes: `DESC.swing`, `DESC_NOTE.swing` (entry/exit signal legend),
  `IDLE.swing`, `TARGET_DEFAULTS.swing`, and a stable slot in the `ORDER`
  preference list. Swing holdings show rank (parsed from the `פריצה #N …` buy
  reason via the existing `#N` rank regex), entry + current price, and P&L.

### FIX 2 — cards no longer overlap at any width
- `.cards` is now a robust grid: `grid-template-columns: repeat(auto-fill,
  minmax(320px,1fr)); gap:16px; align-items:start;` (+ `min-width:0` on `.card`).
  Each card owns its own cell and sizes to its own content, so variable card
  heights can never bleed into a neighbour.
- Verified in headless Chromium with pairwise bounding-box overlap tests at
  **1920px (3 cols), 1024px (3 cols), 390px (1 col)** → **0 overlaps**, 0 JS errors.

### FIX 3 — A/B quality-track badge per screener_track ticker
- **Per-ticker track (A/B) is NOT in the data yet.** `portfolios.json` screener
  positions carry no `track` field, and `screener_picks.yaml` is still a flat
  `positions:` list (no per-ticker `track`/`rank`). **paper-trader must propagate
  the per-ticker track into `portfolios.json` positions** (or publish the enriched
  picks list) for the badge to light up.
- The dashboard **already displays the badge gracefully**: `trackOf()` reads a
  structured `position.track` ("A"/"B") first, else parses a `track/מסלול A|B` from
  the reason; `holdingsHTML` renders `מסלול A` (green) / `מסלול B` (amber) next to
  each screener_track holding, and `pendingHTML` appends `· מסלול X` to pending
  tickers — **only when the field exists, hidden otherwise**. The Hebrew A/B legend
  (`DESC_NOTE.screener_track`) already explains them. Applied to screener_track
  only. Verified by injecting `track:"A"/"B"` into a local copy → badges render
  (e.g. `CELH → מסלול A`, `RBLX → מסלול B`); with the real data they stay hidden.
