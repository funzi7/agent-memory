## The new reader (twitterapi.io)
- `fetch_posts` does ONE GET to
  `https://api.twitterapi.io/twitter/user/last_tweets?userName=<account>` with the
  secret **`TWITTERAPI_API_KEY` in the `X-API-Key` header** (read from env, scrubbed
  from every logged URL/error). `api_base`/`api_endpoint` are overridable via the
  `twitter_track` config block.
- `_parse_twitterapi` parses the JSON `tweets` array (located defensively: top-level
  `tweets`, else nested `data.tweets`) into the internal post shape
  `{id, text, timestamp, author_handle, is_retweet}`.
- **Retweets excluded defensively** (twitterapi.io documents no single flag):
  a populated `retweeted_tweet`/`retweeted_status`, a `type`/`referenced_tweets`
  repost marker, an author handle ≠ the account, or `RT @` text all exclude the
  tweet. `retweetCount` is engagement on the account's OWN tweet — not a
  self-retweet signal, so it's ignored.

## Graceful failure (distinct from "0 posts")
Missing key / HTTP / network / quota / malformed body → `None` with a key-scrubbed
note that starts `twitterapi.io read failed: <reason>`; the tracked list is left
UNCHANGED. A 200 with 0 original posts returns `[]` (no error) — never confused with
a read failure. Dedupe cursor + classifier-failure retry are unchanged.
