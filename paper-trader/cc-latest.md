## What this change did
Fix so the **newly-added `swing` strategy's portfolios get published** even when the
daily cycle finds "no new bar". Opened **PR #7** to `paper-trader` main (Codex

## Why swing was missing from the dashboard
- Swing's CODE is on `paper-trader` main (PR #4 + #6): `swing.py`,
  `sp500_universe.py`, config block, registered → 18 portfolios. 449 selftest
  checks pass. **It was never a missing-code problem.**
- But the published `state/portfolios.json` had only **16** portfolios
  (`last_processed_date: 2026-06-24`). Swing merged **2026-06-25**, after that
  publish. The daily cycle **early-exits at the "no new bar" date gate**
  (`prev_processed == date_str`) **before `save_state`**, so on any day without a
  fresh completed bar the swing portfolios (already initialized in memory by
  `build_or_load`) were never written. Re-running `daily` did nothing.

## The fix (minimal, additive — `main.py`)
At the date gate, before returning:
- `merge_new_portfolios(saved_raw, portfolios)` finds portfolios in config but
  **missing from saved state**.
- If any, `write_state_raw()` publishes them: existing entries kept
  **byte-for-byte** (reused from loaded raw, never round-tripped), the
  freshly-built ones (full cash, no positions/pending) appended, and
  **`last_processed_date` left UNCHANGED**.
- **No `run_day`, no fills, no look-ahead, no double-processing.** `trades.csv` /
  `closes.json` untouched. The genuinely-new-bar path already initializes + trades
  new portfolios, so this only covers the "stuck on the same date" case.
