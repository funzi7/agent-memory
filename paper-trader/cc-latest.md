## What this change did (dashboard display — `paper-trader/app.js`, `sw.js`)
Two screener_track display fixes + a service-worker cache bump so returning visitors
actually receive them. READ-ONLY page; no data or paper-trader code touched.

### FIX 1 — order holdings/pending by the STRUCTURED `rank` field (never alphabetically)
- Holdings already sort by `positions[tk].rank` ascending (both the dashboard card and
  the `strategy.html` detail page go through the one `holdingsHTML`, tiebreak
  alphabetical, absent-rank keeps order). screener_track now renders
  **NVDA → CELH → TSM → FSLR → ADBE → PODD → RBLX** (ranks 1..7) instead of A→Z.
- **New:** the pending affordability preview (`affordability()`) now takes the rank from
  the **structured `o.rank` field** when present, falling back to reason-string parsing
  only for older data — so pending is rank-ordered from the field, no reason parsing, per
  the rule. Structured pending (swing/pullback) already used `o.rank`.
- General: any portfolio whose holdings/pending carry a numeric `rank` sorts by it;
  momentum_scan/leveraged_momentum/swing positions currently have `rank:null` in the JSON
  so they keep their existing order (no reason-parse fallback for the sort).

### FIX 2 — conditional A/B track badge on screener_track (already present, verified)
- A screener_track holding with `track:"A"|"B"` shows a small chip next to the ticker:
  **green "מסלול A"** (`.badge.track-A`) / **amber "מסלול B"** (`.badge.track-B`), via
  `trackOf(q)` → `trackBadge()`. Data-conditional: absent `track` → no chip, no error.
  The Hebrew מסלול A / מסלול B legend on the card is unchanged.
- `track` is not in the live JSON yet (lands with tonight's daily); the badge appears
  automatically once it does. Pending shows the track as a data-conditional text label
  ("· מסלול A"); the styled chip is on the holdings row (the legended surface).
- Bidi per RTL-RULEBOOK: the ticker is Latin-isolated by `applyRtl`, the row right-aligns.

### SW cache bump — the reason a stale client still saw A→Z
`sw.js` is cache-first for the shell (app.js included), so a returning client kept serving
the previously-cached app.js. Bumped **`pt-dash-v18` → `pt-dash-v19`** so `activate` drops
the old caches and every client fetches the current app.js (rank order + badge) on next
visit. HARD reload / bump both done.
