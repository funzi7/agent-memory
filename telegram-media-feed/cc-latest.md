## Why this round existed — build fa3b926d physically FAILED (complete diagnostic)

Telegram Android 9.6 / Chromium 149, ordinary vertical posts, untruncated report. Observed: several seconds of loading, then the video stayed stopped; manual Play required — worse than the previous Play-flash.

**Confirmed failure 1 — source replacement aborted a valid trusted start.** First transition targeted `video-3` (media 1462). The trusted release worked: `userActivationIsActive=true`, `playCalledBeforeReturn=true`, `authenticated-direct` source, the element emitted `play`/`waiting`/`loadstart` with playback legitimately pending while the direct source buffered. The implementation then waited for cache/source preparation and at ~5.09 s invalidated the attempt, paused, released ownership, removed the attached source, called `load()`, changed source generation, attached `retry-direct`, and received `AbortError` — destroying its own activation-bearing attempt. At ~6.84 s a later fallback on the SAME `video-3` element produced `play`/`playing`/resolved promise/advancing time even though `userActivationIsActive=false` — key evidence: the once-unlocked element keeps playing later without fresh activation.

**Confirmed failure 2 — a NEW element was policy-blocked.** A later post used `video-6` (media 1460). At release: activation false, `playCalledBeforeReturn=false` (`not-ready`); the element mounted afterwards, loaded to `readyState=3→4` with valid duration, buffered, `muted/defaultMuted/autoplay/playsInline` all true — and the delayed fallback `play()` rejected with `NotAllowedError` (`policy`), no `playing`, currentTime 0. The ~4 s wait before revealing Play created the user's "thinking then stopping" experience.

**Architectural conclusion:** per-post media elements were the core remaining problem. Telegram Android permits/unlocks ONE element during the initial successful activation; later newly created elements are policy-blocked. No amount of delayed retries, readiness timeouts, source fallbacks, activation conditions, or trusted-release candidate rules fixes that.

## Implemented architecture — ONE persistent session player

### Controller (`lib/persistent-player.ts`, `PersistentFeedPlayerController`)

- Owns the single node's lifecycle: `attach` (once; `session-player-created` with stable `persistentPlayerNodeId`), `beginViewing` (synchronous through the single `play()` call), `endViewing("image"|"album"|"no-post"|"reset")` (pause + hide, KEEP node and source, no `load()`), `playFromUser`/`pauseFromUser` (durable per viewing), `retryFromTrustedGesture` (requires only `event.isTrusted`; needed only while never-unlocked/policy-rejected), `setSuspended` (scrub/visibility/web-app/pagehide; resume replays without activation), `setMutedFromUser`, `handlePlaying/Pause/Ended`, `notifyReadiness` (only completes one armed external-AbortError retry), `handleTerminalSourceError` (ONE bounded failover to the prepared backup, else terminal), `detach` (viewer change/unmount: clears src + releases lease fail-closed; node itself retained and re-attachable).
- **Unlock-once:** `session-player-unlocked` after the node's first genuine `playing`; afterwards every transition plays with no gesture and no `navigator.userActivation` gating (traces prove genuine gestures can report `isActive=false`).
- **One locked source per viewing:** selection order memory Blob → cache Blob → authorized direct URL; attached once (`video.src=` only — an extra `setAttribute("src")` would re-run the load algorithm and self-abort); the previous lock's lease is released only AFTER the new source is attached; slow loading/waiting/stalled/pending-play never replace it; `retry-direct` is gone from the ordinary path. Stale play resolutions/rejections from an older source generation are ignored (`stale-player-event-ignored`).
- Mute: first automatic session playback starts muted regardless of stored preference; explicit session unmute (`setMutedFromUser(false)`) carries to later sources when the browser permits, with exactly one truthful muted fallback if an unmuted start is refused. Unexpected internal pauses get at most 2 bounded resumes per viewing (unlocked node needs no activation).

### Feed integration (`app/feed-page.tsx`)

- `PersistentFeedPlayer` component (feed-level, rendered ONCE, no key): hosts the `<video data-persistent-player-node=…>` inside `.persistent-player-host` — a stable absolute layer in the feed scroller positioned over the active post's slot via `translateY(index*100%)` (`z-index:1`; post overlays are `z-index:2+`). Carries the whole video UI (tap target, scrubber, spinner, seek flash, fullscreen dup controls, watch/completion accounting attributed per media with refs reset on media change and final flush attributed to the departed media).
- Ordinary single-video posts render a `persistent-video-slot` placeholder in `MediaRenderer` (their warm-prep effect is skipped); the post card bridges controls visibility/pinning/rail export/double-tap Like via a slot adapter (`onPersistentSlotAdapter`). Albums keep the entire previous per-item FeedVideo/coordinator/playback-group architecture, separated and documented; the shared group's committed target still prevents album items and the persistent player from playing simultaneously.
- Feed-level prepared-source registry: `acquireAuthorizedStored` (with HEAD revalidation) for the playback post and ±1/+2 neighbors; leases wait in a map; `selectPersistentViewingSource` takes one synchronously at viewing start or falls back to the direct URL. A lease materializing during the active viewing is recorded `late-source-ignored` and retained for the next viewing.
- Viewing transitions run in a LAYOUT effect, so flushSync-driven trusted releases reach `play()` inside the gesture task (latency + first-unlock bonus); wheel/settlement/restore transitions play identically without gestures. The trusted-release deterministic targeting + programmatic snap + pending-snap settlement deferral from round 5 is retained for snap control and diagnostics; its play-path now only calls `retryFromTrustedGesture` when the node is never-unlocked and policy-rejected.
- Viewer identity change (identityType/effectiveAccessToken) bumps `viewerEpoch` → controller detach (src + leases cleared fail-closed) → re-attach same node. Feed refresh/mode/topic-filter/likes/progress never recreate the node.
